## Changes

### `backend/database.py`

`_migrate()` now also adds four observability columns (additive, idempotent):

- `enriched_at TEXT DEFAULT ''`
- `enrichment_model TEXT DEFAULT ''`
- `enrichment_prompt_version TEXT DEFAULT ''`
- `enrichment_input_hash TEXT DEFAULT ''`

(`enrichment_error` was already in the migration list; will get applied on next startup of any DB that hasn't seen it.)

### `backend/storage/news.py`

1. **`_bulk_upsert_articles()` upsert SQL** now uses `COALESCE(NULLIF(excluded.col, ''), col)` for the content-bearing columns: `scraped_content`, `ai_summary`, `title_zh`, `content_zh`, `tags_zh`, and the four new observability columns. `enrichment_attempts` uses `MAX(old, new)`.

   Fixes the latent bug that a Stage A re-fetch could blank out the Chinese fields already populated by Stage B. Cross-agent handoff note in the plan called this out explicitly.

2. **`list_pending_enrichment()`** rewritten:
   ```sql
   WHERE enrichment_status != 'failed'
     AND COALESCE(enrichment_attempts, 0) < ?
     AND (COALESCE(title_zh, '') = '' OR COALESCE(content_zh, '') = '')
   ```
   Content is the source of truth, status is hint. A stale `pending` row with full Chinese content is no longer re-enriched.

3. **New `repair_enrichment_status()`**: one-shot SQL UPDATE that
   - flips `enrichment_status='done'` (and clears `enrichment_error`) for rows with non-empty `title_zh` + `content_zh`
   - flips `enrichment_status='failed'` for rows past `MAX_ENRICHMENT_ATTEMPTS` with incomplete content
   - never touches the Chinese content columns themselves

   Re-exported via `backend/storage/__init__.py`.

### `scripts/repair_enrichment_status.py`

CLI wrapper around `repair_enrichment_status()`. Supports `--dry-run` (count-only preview). Idempotent: subsequent runs report 0 changes.

### `.claude/rules/backend.md`

Updated DB Schema section to reflect the four new columns, the content-as-source-of-truth status semantics, and the upsert preserve-on-empty pattern.
