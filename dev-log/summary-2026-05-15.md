## Feature 1 — State repair + schema hygiene

DB observation: 2000/2000 rows marked `pending` while 1364 had complete `title_zh + content_zh`. Live schema missing `enrichment_error`.

- `_migrate()` adds 4 observability columns: `enriched_at`, `enrichment_model`, `enrichment_prompt_version`, `enrichment_input_hash`.
- `_bulk_upsert_articles` switches content-bearing columns to `COALESCE(NULLIF(excluded.col,''), col)` — Stage A re-fetch can no longer clobber Chinese fields. `enrichment_attempts` uses `MAX(old, new)` for monotonicity.
- `list_pending_enrichment()` gates on content completeness + retry ceiling, not status alone.
- New `repair_enrichment_status()` reconciles status → content. Idempotent.
- `scripts/repair_enrichment_status.py` — one-shot CLI with `--dry-run`.

Verified on live DB copy: 1364 marked done as predicted. Post-repair mismatch count = 0.

## Feature 3 — MiniMax wrapper reliability

`call_minimax()` was a black box: any failure looked the same to the caller. Now:

- Exception hierarchy: `MiniMaxConfigError` / `MiniMaxRateLimitError` / `MiniMaxHTTPError` / `MiniMaxTimeoutError` / `MiniMaxResponseError`.
- `call_minimax` raises (no more silent fallback); internal retry for 429 / 5xx / timeout / network at 1s → 3s → 10s backoff. 4xx non-retryable. Empty content rejected as `MiniMaxResponseError`.
- `time.sleep(3)` removed from inside the function; `services.enrich_pending_news` sleeps `MINIMAX_RATE_LIMIT_SLEEP_SEC` *between* iterations. One-off brief generation no longer paid that hidden cost.
- Caller handling:
  - services: `MiniMaxConfigError` → abort cycle without burning attempts. Other `MiniMaxError` → flip to `failed` only at attempts ceiling.
  - news_briefs: `MiniMaxConfigError` re-raises (operator-fixable); other `MiniMaxError` falls back to rule-based brief.

9-case mocked unit tests cover all exception paths.

## Feature 2 — Article enrichment v2 (JSON)

Old line-prefix `TITLE_ZH:` format was brittle. Replaced with strict JSON contract:

```json
{
  "title_zh": "...",
  "summary_zh": "...",
  "content_zh": "...",
  "tags_zh": ["移民签证"],
  "category": "Sociedade",
  "relevance_reason": "..."
}
```

- `ARTICLE_ENRICHMENT_PROMPT_VERSION = "article_enrichment_v2"` persisted to every enriched row.
- New parser tolerates Markdown code fences and surrounding chatter; validates types; filters `tags_zh` to the allowed 10-tag set; falls back to legacy line-prefix parser on malformed JSON for rollout safety.
- New schema columns: `summary_zh`, `relevance_reason` (preserve-on-empty).
- `re_enrich_article` writes the four observability columns: `enriched_at` (only when both `title_zh` and `content_zh` are non-empty), `enrichment_model`, `enrichment_prompt_version`, `enrichment_input_hash` (sha256 of title+input, first 16 hex).

9-case parser unit tests pass.

## Feature 5 — Frontend preview semantics

- Card / Featured / HomePage lead / HomePage grid / NewsTab featured: `summary_zh || content_zh || ai_summary || summary`.
- Article modal body: `content_zh || summary_zh || ai_summary || summary` (modal has room for the fuller body).
- "Original" section unchanged: `scraped_content || summary`.

Legacy rows without `summary_zh` keep rendering via the fallback chain. Safe to ship before backfill.

## Feature 4 — Brief input selection + generated_by

Old selector accepted any non-empty `tags_zh`; a recent brief referenced 170 articles.

- Selection now requires `enrichment_status='done'` AND (`summary_zh` OR `content_zh`) non-empty. China brief additionally requires non-empty `tags_zh`.
- China candidates ranked by (priority-tag count, total-tag count, published-desc). Priority tags: `移民签证, 法律法规, 工作就业, 税务财务, 安全治安`.
- Portugal ranked by published-desc.
- Dedupe by first-8 word-char prefix of normalized title (catches "same story (updated)" cross-source dup).
- Hard caps: `MAX_CHINA_BRIEF_ARTICLES = MAX_PORTUGAL_BRIEF_ARTICLES = 20`.
- `_article_digest` uses `summary_zh` to keep the 20-article prompt small.
- New `daily_news_briefs.generated_by` column (`'llm' | 'fallback' | 'failed'`) tracks provenance; idempotent migration.

## Feature 6 — Admin diagnostics
